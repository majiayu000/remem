## High

### H1 - Local app request host trust needs hardening

**Fact:** The local Node app has startup and request checks, but the audit found
a request-host trust gap that should be closed before publishing exact
reproduction details.

**Impact:** A browser-facing local app without HTTP auth should treat host and
origin metadata as untrusted. If request-host validation is incomplete, local
memory data and write routes can be exposed through browser-mediated requests.

**Confidence:** High. Detailed reproduction notes should stay out of public
docs until the remediation ships.

**Fix:** Validate request host metadata against the bound loopback address and
port, add a per-session CSRF nonce or local API token for write routes, and add
negative regression tests for non-loopback host metadata.

### H2 - Staleness/source-anchor failures are downgraded to `untracked`

**Fact:** API helpers catch source-anchor lookup errors, log `warn`, and return fallback labels (`src/api/helpers.rs:16`, `src/api/helpers.rs:37`). The lossy helper maps per-memory errors to `memory_staleness_label`, whose default anchor is `untracked` (`src/memory/staleness.rs:45`, `src/memory/staleness.rs:101`). Similar lossy paths exist in search explain (`src/retrieval/search/memory/text.rs:571`) and context rendering (`src/context/query.rs:123`, `src/context/prompt_submit.rs:191`).

**Impact:** A user-visible API/search/context response can return 200 OK with a plausible but wrong `source_anchor`. This violates the repo's no-silent-degradation rule for user-visible memory correctness and can hide cases where a memory should be treated as stale or verify-before-trust.

**Confidence:** High.

**Fix:** Use strict staleness calculation at public/API boundaries. If anchor lookup fails, return a structured error or an explicit `source_anchor="error"` label with error-level logging. Keep lossy behavior only in deliberately best-effort diagnostic views.

### H3 - Manual/API save accepts unvalidated memory shape

**Fact:** `save_memory_inner` directly defaults and forwards `project`, `title`, `memory_type`, `scope`, and `req.text` without validating type/scope/text (`src/memory/service/save.rs:62`). Legal memory types are separately enumerated in `MEMORY_TYPES` (`src/memory/types.rs:178`), and architecture docs define scope as only `project` or `global` (`docs/ARCHITECTURE.md:351`). The durable insert/update writes these strings directly to `memories` (`src/memory/store/write.rs:217`, `src/memory/store/write.rs:370`), while the baseline schema only stores them as unconstrained text (`src/migrations/v001_baseline.sql:74`).

**Impact:** A typo such as `memory_type="decison"` or `scope="globla"` can become an active memory that search, context, ownership, stats, and lifecycle code interpret inconsistently. Whitespace-only `text` can also pass this boundary, reintroducing an empty-memory class that previous audits called out as high-risk.

**Confidence:** High for type/scope drift, medium-high for empty text based on boundary inspection.

**Fix:** Add a typed validation layer before local-copy or durable writes: non-empty trimmed text, known `MemoryType`, known scope, and normalized strings. Return typed validation errors to REST/MCP callers. Consider DB `CHECK` constraints after existing data is cleaned.

## Medium

### M1 - Raw archive fallback failure is swallowed as an empty result

**Fact:** Curated searches below the threshold call raw archive fallback (`src/memory/service/search.rs:6`, `src/memory/service/search.rs:48`). If `search_raw_messages` fails, the code logs `warn` and returns `vec![]` (`src/memory/service/search.rs:132`).

**Impact:** The caller cannot distinguish "no raw archive hits" from "raw archive query failed." That can hide recall degradation exactly where fallback is meant to recover unpromoted conversation content.

**Confidence:** High.

**Fix:** Return `Result` from `maybe_fallback_raw`, or expose a structured `raw_hits_error` in search responses. Log at error level when the fallback is supposed to affect user-visible recall.

### M2 - Raw archive preview is exposed by default in the plugin app search surface

**Fact:** The plugin design says the app UI must not expose raw archive content by default without user action (`docs/spec-codex-plugin-complete-design.md:352`). Current tool metadata advertises raw archive fallback on the ordinary search tool (`plugins/remem/apps/remem/tools.js:36`), and the widget merges `payload.raw_hits` into normal result rows (`plugins/remem/apps/remem/public/widget.js:184`) and renders preview/content in detail (`plugins/remem/apps/remem/public/widget.js:227`).

**Impact:** Raw chat history can contain sensitive prompts, paths, and operational context. Combined with H1, this becomes a larger local data exposure path.

**Confidence:** Medium-high. Whether raw hits appear depends on backend search conditions, but the default UI path is present.

**Fix:** Require an explicit `include_raw` or "show raw archive" user action. Render raw hits in a separate clearly marked section and add tests that ordinary search does not expose raw preview by default.

### M3 - Schema currentness checks only validate the v022 drift case

**Fact:** Hook DB open calls `open_db_no_migrate`, then `validate_schema_invariants` (`src/db/core.rs:143`). `ensure_schema_current` checks migration records and version numbers but not actual objects (`src/migrate/run.rs:42`). `validate_schema_invariants` only checks v022 objects (`src/migrate/schema_drift.rs:7`). The schema drift test constructs exactly that v022-missing case (`src/migrate/tests_schema_drift.rs:43`), and the convergence table whitelist omits many later migration objects (`src/migrate/tests_convergence.rs:22`).

**Impact:** If `_schema_migrations` claims a current schema while later objects are missing, hook/no-migrate paths can accept the DB and fail later in user-visible capture/search/context paths.

**Confidence:** High for coverage gap, medium for live likelihood.

**Fix:** Replace the v022-only invariant with a declarative registry of tables, columns, indexes, triggers, and FTS objects introduced by each migration. Add tests for a post-v022 drift case such as a marked-applied v045 with missing usage columns/tables.

### M4 - `remem encrypt` treats an existing `.key` file as proof of encryption

**Fact:** `run_encrypt(false)` returns success as soon as `.key` exists (`src/cli/actions/maintenance.rs:43`). Normal DB open does validate key readability (`src/db/crypto.rs:70`), and install runtime does try to migrate using the existing key (`src/install/runtime.rs:145`), but the manual encrypt command never reaches `encrypt_database` when a key file already exists (`src/db/crypto.rs:229`).

**Impact:** A leftover or manually created `.key` with a plaintext DB causes `remem encrypt` to report "already encrypted" without completing the migration. Users get a false-success recovery signal.

**Confidence:** Medium.

**Fix:** When `.key` exists, verify the DB can be opened as encrypted. If not, detect whether the DB is plaintext and return a clear repair path or guarded force mode. Add tests for `.key` present plus plaintext DB.

### M5 - Release manifest `asset.file` is not restricted to a safe basename

**Fact:** The npm installer verifies SHA-256 format but accepts arbitrary `asset.file` strings (`npm/remem/scripts/install.js:97`) and joins them into a temp archive path (`npm/remem/scripts/install.js:138`). The plugin runtime follows the same pattern (`plugins/remem/scripts/remem-runtime.js:322`, `plugins/remem/scripts/remem-runtime.js:399`).

**Impact:** A compromised or malformed manifest can use `../` or path separators to escape the intended temp archive filename before checksum validation. The checksum still protects binary integrity, but local path handling and URL construction are too permissive for an installer boundary.

**Confidence:** High.

**Fix:** Validate manifest file names against exact platform basenames such as `remem-linux-x64.tar.gz`, reject `/`, `\`, and `..`, and download to a fixed local filename independent of the manifest name. Add malicious filename tests to both npm and plugin runtime suites.

### M6 - Save API maps most service errors to 500 via string matching

**Fact:** `handle_save_memory` returns 400 only when the error string contains `outside the allowed directory`; everything else becomes `500 save_failed` (`src/api/handlers/save.rs:73`). Service tests already cover input-like failures such as out-of-bounds and directory local paths (`src/memory/service/tests.rs:289`, `src/memory/service/tests.rs:443`).

**Impact:** New validation failures will default to internal-server errors, and clients cannot reliably distinguish user input problems from storage/runtime failures.

**Confidence:** High.

**Fix:** Introduce typed errors such as `Validation`, `LocalPath`, `Storage`, and `Internal`, then map them to stable HTTP status codes and error codes.

### M7 - Web API "read-only" spec conflicts with detail GET access tracking

**Fact:** `SPEC-web-api` says the web API is GET-only and read-only (`docs/specs/SPEC-web-api.md:1`, `docs/specs/SPEC-web-api.md:25`). `GET /api/v1/memories/{id}` calls `mark_memories_accessed` (`src/api/handlers/detail.rs:50`).

**Impact:** Detail panel loads, retries, and prefetches mutate `last_accessed_epoch/access_count`. That may be desired telemetry, but it violates the current spec and can skew usage signals.

**Confidence:** High.

**Fix:** Either remove access tracking from GET detail, or update the spec/API docs and tests to declare this as intentional telemetry. If it remains, consider moving it behind an explicit POST.

## Low

### L1 - npm publish workflow lacks a real postinstall/bin smoke

**Fact:** Release workflow runs version sync and installer resolver tests before publishing (`.github/workflows/release.yml:112`, `.github/workflows/release.yml:114`). `package.json` has a `postinstall` download and `test` script that executes `node bin/remem.js --version` (`npm/remem/package.json:15`), but the publish job does not install the packed package and run the final binary. Current installer tests are resolver-level (`npm/remem/scripts/install.test.js:17`).

**Impact:** A broken release manifest, tarball layout, postinstall download, vendor path, or bin wrapper could still be published.

**Confidence:** High.

**Fix:** Before `npm publish`, run `npm pack`, install the generated tarball in a temp project, execute `remem --version`, and verify the reported version and checksum source.

### L2 - Public API integration coverage is thin

**Fact:** The crate publicly exports API token helpers, router construction, server start, and `DbState` (`src/api.rs:9`). The external integration test file currently has one test for token helper setup (`tests/api_public.rs:49`).

**Impact:** Router/auth/error envelope/save/search behavior is mostly protected by crate-internal tests, so externally visible API contracts can drift without an integration-level failure.

**Confidence:** High.

**Fix:** Expand `tests/api_public.rs` to cover authenticated router status/search/list/detail/save, invalid save responses, and the intended read/write contract.

## Healthy Areas Observed

- The normal SQLCipher open path is fail-closed: a missing key without explicit plaintext override is rejected, and an applied key must read schema (`src/db/crypto.rs:60`, `src/db/crypto.rs:70`).
- Migration execution is transactional and logs rollback failure at error level (`src/migrate/run.rs:9`).
- Plugin version sync is currently clean at `0.5.97`.
- Node local app already has body-size limits, JSON-only POST checks, cross-site
  `Sec-Fetch-Site` checks, and loopback CLI binding. H1 is specifically about
  request host trust, not total absence of local-app hardening.
- Runtime downloads require SHA-256 metadata and reject checksum mismatches. M5 is about safe manifest filename handling before download/extract.
