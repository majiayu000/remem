## Proposed Design

Add nullable review metadata columns to `memory_candidates`, then route all
single and batch outcomes through metadata-aware helpers. Batch commands first
resolve filters into a preview, then re-resolve the same filter inside one
transaction so the executed id set is generated by the same code path. Queue
health uses aggregate SQL over indexed candidate fields; status, doctor, CLI,
and REST consume those shared readers rather than duplicating logic.

## Data Flow

CLI and API filters resolve against pending `memory_candidates`. Approve paths
load candidate rows, route them into memory writes with unchanged promotion
semantics, then update candidate review metadata. Discard paths update only the
candidate review row. Metrics read candidate creation, review status,
reviewed-at, updated-at, project, and block-reason fields.

## Alternatives Considered

- Infer review actor from promoted memory operations: rejected because discard
  rows and batch runs need direct per-candidate provenance.
- Separate preview and mutation queries: rejected because filter drift could
  approve/discard a different set from the preview.
- Add web UI now: rejected by the accepted product non-goal.

## Risks

- Security: user-provided filters must remain parameterized SQL values.
- Compatibility: existing databases need additive nullable columns only.
- Performance: status and doctor must avoid per-row scans for normal health
  checks.
- Maintenance: CLI and REST filters should stay aligned through shared review
  query semantics.
